1. Boris Cherny：Claude Code 之后，写代码正在变成”管理 Agent”

来源: 宝玉的分享
原文链接: Boris Cherny：Claude Code 之后，写代码正在变成”管理 Agent”
推荐指数: ⭐⭐⭐⭐⭐ (5/5) - 必读，这是 Claude Code 创造者的第一手访谈，揭示了编程工具革命的核心趋势

核心概要:
这篇文章是 Anthropic 工程师 Boris Cherny 在 Sequoia 2026 年 AI Ascent 大会上的访谈实录，Boris 是 Claude Code 的创建者。访谈的核心观点是：编程正在从”写代码”转向”管理 AI Agent”。Claude Code 从一个三人小团队的孵化项目，发展成为 2026 年初年化营收超过十亿美元的产品，成为 Anthropic 史上从预览到十亿美元产品最快的一次成功案例。

文章深入探讨了几个关键命题：Boris 个人在 2026 年完全停止了手写代码，每天合并几十个 PR；Claude Code 早期半年没有产品市场匹配（PMF），直到 Opus 4 发布后才迎来指数增长；他现在通过手机运行 5-10 个常驻 session，每个 session 管理几百个 Agent，晚上还有几千个 Agent 在后台运行深度任务。Boris 提出历史类比：软件构建会像识字一样普及，最合适写会计软件的是会计师而不是工程师，因为编程是简单部分，懂业务才是难的部分。

文章还讨论了 SaaS 行业的护城河变化：AI 会抹平切换成本和流程效力这两种护城河，但网络效应、规模经济、独占资源仍然有效。Boris 认为 Anthropic 的真正领先不在技术模型，而在组织流程——整个公司已经没有手写代码，所有 SQL 和产品代码都由模型生成，员工的 Claude 之间通过 Slack 互相沟通。

关键洞察:

  1. 产品悬置（Product Overhang）策略：Anthropic 提前为下一代模型做产品开发，即使早期产品不具备 PMF 也持续推进。Claude Code 前半年几乎没人用，Boris 自己也只用它写 10% 的代码，但他们赌的是模型能力会涨到一个点。这种”为未来模型做开发”的策略与典型的”先验证需求再做产品”的 SaaS 逻辑完全相反。

  2. 从手写代码到管理 Agent 的核心工作流变化：Boris 的工作模式是 Loop 调度——让 Claude 通过 cron 起定时任务循环，自动盯着 PR、修 CI、rebase、保持测试健康、从 Twitter 拉反馈并聚类整理。他几十个 Loop 一直在跑，自己只在 Slack 上接收报告。这是未来工作流的核心范式。

  3. 通才崛起与角色边界模糊：当写代码的边际成本接近于零时，原本被排除在工程之外的角色（财务、设计、产品经理、数据科学家）都有了直接出工程产物的能力。Claude Code 团队里，工程经理、产品经理、设计师、数据科学家、财务、用户研究员，每一个人都在写代码。

  4. SaaS 护城河的结构性变化：AI 能帮用户从一个工具迁到另一个工具，一夜之间迁移完原本复杂的工作流配置（切换成本消失）；Claude 4.7 能 hill-climb 任何流程，自己迭代优化直到达成目标（流程效力消失）。但网络效应、规模经济、独占资源依然成立。

  5. 组织流程改造比技术领先更重要：模型层基本没差距（内部用的 Mythos 和 Opus 4.7 未来也会对外开放），但产品层有更大差距，原因来自流程。要把整个公司从手写代码切到模型生成代码，要让员工的 Claude 之间通过 Slack 互相问问题，这是组织行为的改造，比技术本身慢得多。

评价:

  • 优点:

    • 第一手资料价值极高：这是 Claude Code 创造者的直接访谈，不是二手分析，信息密度和可信度都很高
    • 结构清晰，要点明确：文章整理了 12 个核心议题和速览问答，便于读者快速抓住重点
    • 洞察具有前瞻性：对编程工具、SaaS 行业、组织形态的预判有明确的判断和时间线
    • 承认局限性：文章中加了多处”注”，指出 Boris 观点的样本偏差、数据偏差、适用范围局限，体现了批判性思维
    • 具体案例丰富：Boris 的工作流、团队协作方式、具体产品功能都有详细描述，不是空泛概念

  • 不足:

    • 样本偏差明显：Boris 用的是 TypeScript+React 这种最受模型偏爱的栈，他的结论”编程已被解决”对 SAP、C++ 老系统、嵌入式系统、高合规场景是否成立存疑
    • 历史类比数据有偏差：Boris 说 15 世纪欧洲识字率约 10%、今天约 70%，实际学界估计是 30% 和 90%，虽然不影响核心论点，但数据准确性有问题
    • 对大企业迁移难度估计不足：文章对 5000 人银行 IT 部门的合规、风控、变更管理、审计追踪等约束讨论不够，这些不是”我会写代码”就能跨过去的
    • 对安全风险讨论不足：印刷术类比只说了正面影响，但没展开印刷术催生的几百年审查制度、版权战争、宗教改革动荡。”软件人人都能写”对应的不只是创造力释放，也包括恶意软件、深伪诈骗、AI 生成漏洞利用的同步爆发

  • 适用场景:

    • 强烈推荐给：AI 工具开发者、技术团队管理者、SaaS 从业者、关注 AI 对生产力影响的投资者
    • 参考价值：普通开发者（理解未来工作流变化）、产品经理（学习产品悬置策略）
    • 谨慎参考：传统企业 IT 负责人（Boris 的经验很难直接迁移到高度合规的环境）、安全研究者（文章对风险讨论不足）
