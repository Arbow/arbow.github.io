文摘：需要自进化的不是 Agent，而是 Harness

May 27 2026 文摘

空谷 Arvin Xu（@arvin17x）在这篇长文中提出了一个核心观点：单独的模型已经不再是产品，模型 + Harness 才是产品。而 Harness 不应是静态的执行环境，它必须能自我进化。

这篇文章来自 LobeHub 的工程实践，作者分享了他们在接入 70+ 模型 Provider 的过程中，如何设计一个「Self-Evolving Harness」——一个能随着使用自动变好的运行时。

一个正在发生的范式转移

2026 年上半年，AI 行业正在经历范式转移：模型能力的边际效应递减，产品竞争从「模型能力」转向「对环境的感知和控制」。

Phil Schmid（最早提出 Harness 概念的人）所说的「持久性（Durability）」——模型在执行数百次工具调用后，还能多好地遵循指令？一个模型在 benchmark 上领先 1%，并不能说明它在第 50 步不会误删文件、忘记约束，或者在工具失败后开始编造结果。

悖论：模型越来越强，产品体验却未必同步提升。因为缺失了一环：产品本身能否随模型迭代、用户使用而自动进化。

什么是 Harness？

如果把模型看作 CPU，上下文窗口像 RAM，Agent Harness 就接近操作系统。它管理启动过程、上下文、工具、错误和生命周期。Agent 则是跑在这套运行时之上的应用逻辑。

在 LobeHub，Harness 围绕几层系统构建：

  • Model Runtime：适配不同模型供应商
  • Agent Runtime：实现云原生的步级 Agent 循环
  • Context Engine：管理上下文策略
  • Environment：提供 Page、Task、Group、IM 等不同使用环境

但 Harness 本身也会过时——模型接口会变，工具 schema 会变，错误模式也会变。所以问题变成：Harness 能不能自己进化？

Tracing 是自进化的核心构件

Harness 需要在四个核心维度上持续优化：上下文策略、工具编排、错误认知、模型适配。

实现前提是一个最基础的能力：可追溯。每一次 Agent 执行——调用了什么模型、每个 LLM 调用花了多少钱、哪一步出错了——都需要被完整记录。

行业困境：为什么 Tracing 这么难？

作者分析了 LangChain、CrewAI、OpenAI Agents SDK、AG2 等主流框架的源码，发现共同模式：tracing 都是在执行模型之上「后加」的。

  • LangChain：callback 是可选的，忘了注册就丢 trace
  • CrewAI：事件监听器挂在事件总线上，事件丢失 = trace 断裂
  • OpenAI Agents：需要显式创建 trace，不自动传播
  • AG2：middleware 是可选安装的，不装就零 tracing

不是因为大家不会用 OTel，是因为所有 runtime 架构都没把 observability 当作一等公民。

LobeHub 的 agent runtime 采用状态机模型，按「单步执行」原则做了细粒度拆分，每个 step 天然就是一个 trace event。Tracing 是执行的副产品，不是后装功能。

Execution Snapshot

为每次 Agent 执行生成完整的执行快照：调用了哪个模型、每个 LLM 请求花了多少 Token、每一步类型和耗时、在哪一步出错、错误是什么。

生产案例：Error Pattern 自动巡检

这是 Self-Evolving Harness 理念的真实落地。

LobeHub Model Runtime 接入 70+ AI Providers，每天大量 Agent 运行。每次模型调用失败都留下错误记录。Error handling 本质上是后置的——你只能在错误真实发生后，才知道需要处理什么。

做法

把失败 trace 的查询权限开放给 Agent，让它批量解析 tracing，按 provider、errorType、status code 和 message 分桶，对照已有 pattern 找出未覆盖的错误，生成修复建议。标准化巡检流程为 7 步：

  1. 数据采集：拉取最近的 error records，多维度分桶
  2. 模式识别：对比已有 ERROR_PATTERNS，识别新错误模式
  3. 自动分类：将用户侧错误与 Harness 自身 bug 分开
  4. 自动修复：用户侧错误直接更新匹配 Pattern
  5. 自动提交：commit → push → 开 PR
  6. 自动清理：删除 Dashboard 中匹配新 pattern 的历史噪音
  7. 根因分析：Harness 自身 bug 深入分析根因，创建修复 Task

成果

系统运行了九轮，数据说明问题：

  • 累计 Pattern 从 31 增长到 104 后趋于平稳，新增 Pattern 持续下降至 0——即使 error 日志量在 334~1158 之间大幅波动，Pattern 库趋于饱和，新错误几乎全部命中已有 pattern
  • Agent 自主发现了 20+ Harness 自身缺陷：Tool schema 不兼容、负数 max_tokens、DeepSeek reasoning_content 丢失、Context Window 过载等
  • Agent 成功率从约 75% 提升到 95% 以上

Harness 自进化的四个层次

  • L1 · 纯人工：人发现问题，人分析，人修复
  • L2 · Agent 辅助：Agent 找出可疑问题，人确认，Agent 执行部分修复
  • L3 · Agent 主导：Agent 完成数据采集、模式识别、代码修改、PR 提交，人保留关键判断
  • L4 · 自主进化：Harness 持续观察自身运行状态，自动优化 Context Engine 策略、调整 Tool schema 兼容层、预测和预防错误

LobeHub 的 Error Pattern 巡检大致处在 L3。L4 是正在实现的方向。

为什么 Consumer Product 本质上是 Self-Evolving 的 Harness？

传统 SaaS：产品 = 功能代码 + 用户数据。功能是固定的，产品变好靠版本更新。

AI-Native 产品：产品 = Harness（运行时）+ 用户交互数据 + 进化能力。每一次用户交互都在为 Harness 提供进化信号。

真正的护城河不是模型，而是 Harness 中沉淀的上下文、pattern、trajectory。模型是通用的、可替换的，Harness 是专属的、持续积累的。

文章也对比了 OpenClaw、Hermes 等自部署方案：虽然技术上出色，但每个用户的 Agent 执行量有限，本地部署使得应用开发方无法获得原始执行数据，反馈信号太稀疏。稀疏信号无法支撑高效的自我迭代，进化天花板一眼望得到头。

Consumer Product 天然拥有信号密度优势。以 LobeHub 为例，每天上万次 Agent 执行，Error 样本按分钟级积累。反馈闭环的速度，直接决定了 Self-Evolving 能进化到什么程度。

The Bitter Lesson Revisited

LangChain 一年内重构了三次架构，Manus 的 Harness 六个月内重构了五次，Anthropic 也在不断从 Claude Code 的 Harness 中剥离过时的逻辑。模型在快速迭代，手工编写的 Harness 逻辑随时可能过时。

唯一的解法：让 Harness 自己能进化，而不是等人来重构。

正在构建的未来

  • Agent Level 自动进化：夜间自动回放当天所有 Tracing，分析失败模式和优化机会，自我调整策略
  • 用户 Level 自动进化：自动识别用户使用模式，更新生成提示词与 Persona 记忆
  • 全局自动化 Eval Harness：每个 Failed Task 自动进入评估→归因→修复闭环

一句话总结：新一代 Consumer Product 的本质，是一个活的、会自我进化的运行时。它观察每一次交互，从错误中学习，自动修复自己，在用户不知不觉中变得更好。
