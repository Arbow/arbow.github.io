文摘：一文搞懂如何在 Codex 中使用 Goals

May 26 2026 文摘

来源: 微信公众号 - AI寒武纪
原文: OpenAI Cookbook - Using Goals in Codex
发布日期: 2026-05-19

核心概要

OpenAI 发布了关于 Codex 中 Goals 机制的官方指南。Goals 是 Codex 里的持久目标机制——不同于一次性的 prompt，它给 Codex 一个持续有效的完成条件：任务什么时候算做完、怎么验证、过程中什么不能动。只要目标还在、还在预算内，Codex 可以自己检查进度、决定下一步，不需要每轮重复目的。

Goals vs 普通 Prompt

特性 普通 Prompt Goals
生命周期 一次性 持久（可暂停/恢复/清除）
完成判断 用户决定 基于证据校验
迭代能力 每轮需重新描述目标 自动继续直到达标
适用场景 单步任务 多步调查、优化、研究

好的 Goal 定义六件事

  1. 结果：工作完成时应该是什么状态
  2. 验证方法：用什么来证明（测试、benchmark、报告）
  3. 约束：什么不能退化
  4. 边界：可以用哪些文件、工具、数据
  5. 迭代策略：每次尝试后怎么决定下一步
  6. 阻塞停止条件：什么时候停下来报告受阻

弱 Goal 例子：

/goal 将 p95 延迟降至 120ms 以下，同时不引起测试回退

强 Goal 例子：

/goal 将 p95 checkout 延迟降至 120ms 以下，以 checkout 基准测试验证，同时保持正确性测试套件全部通过。仅使用 checkout 服务及相关测试。每次迭代之间，记录变动内容、benchmark 结果，以及下一步要尝试的最佳实验。如果 benchmark 无法运行或无有效路径剩余，则停止，并附上已尝试路径、收集到的证据、阻塞点，以及所需的下一步输入

Goal 激活后的三个变化

  1. 目标持续可见 — 即使跑了几轮测试失败了，原始目标还在
  2. 从空闲线程继续 — 只在线程空闲、目标激活、在预算内时才继续
  3. 完成必须有证据 — 不能因为”大概差不多了”就标记完成

内部设计要点

  • Goal 是线程范围的状态，不是全局记忆，也不是项目级指令
  • 继续是事件驱动的，不是简单循环——只在安全边界处检查
  • 纯规划工作不触发继续，被打断会暂停目标
  • 达到预算上限时停止工作、汇总进度，不等于完成目标

研究场景示例：复现论文

文章用 Buehler 等人的 Deep Hedging 论文做了示范。强 Goal 不是笼统的”复现这篇论文”，而是：

尝试每一个主要结果，验证输出，最终形成一份报告，将复现成功的工作机制、近似训练结果、被阻断的精确复现路径及剩余不确定性区分开来说明

最终报告对每个结论逐一审计：有的可以近似复现（重建了定价机制、训练了策略），有的因缺少源数据（原始训练路径、种子、检查点）无法精确重现。Goal 的价值在于让工作在遇到阻塞后继续推进，同时让最终语言保持诚实。

什么时候不该用 Goals

  • 一行代码的修改、简单解释、短代码评审
  • 终点线模糊的任务（”把这个做得更好”没有可靠完成条件）
  • 不确定性未声明的场景

Goals 在三个特性同时具备时最有力：有持久目标，有基于证据的完成线，路径可能需要多轮调查。

评价

这篇文章是对 OpenAI 官方文档的优质中文解读，把 Goals 的设计理念和使用方法讲得很清楚。最有价值的部分是”弱 Goal vs 强 Goal”的对比和复现论文的研究场景示例——它展示了 Goals 不只是让 AI 自主执行，更重要的是定义”做完”意味着什么。证据分级审计的思路（精确复现 vs 近似结果 vs 受阻路径）对科研场景非常实用。对于使用 Codex 或类似 AI 编程工具的开发者来说，这是一篇值得收藏的实践指南。
