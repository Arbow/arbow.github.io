文摘：一文读懂 Harness Engineering——让 AI 不再离经叛道的壳

Apr 02 2026 文摘

文章信息

来源: 腾讯科技
作者: Yousa、博阳
原文链接: 一文读懂 Harness Engineering

核心观点

2026年 Q1，大模型应用层最具统治力的热词是 「Harness」。LangChain 的实证数据显示：仅更换 Harness 架构，模型在 Terminal Bench 2.0 上的通过率从 52.8% 提升到 66.5%，底层模型权重一个字节未改。

但 Harness 不是点石成金的魔法，而是一套工业级管理制度——让 Agent 从「金鱼」变成能跑长途的「汽车」。

Harness 三层架构

第一层：从记事本到管理制度

核心问题：Agent 记忆像金鱼，上下文窗口满了就忘。

演进路径：

  • AutoGPT (2023.03)：空白记事本，模型自己管理记忆
  • Devin (2024.03)：结构化 Planner 面板，可视化进度条
  • Claude Code (2025.02)：CLAUDE.md + scratchpad，业内最广泛模仿的范式

Anthropic 的 Context Engineering：

  • 提高上下文效率：System Prompt 当代码维护、工具描述精确化、外部存储（RAG）
  • 上下文压缩：对话摘要、滑动窗口策略、删除无用工具返回

关键认知跃迁：从「做更好的记事本」转向「围绕严格遵守工作流程，构筑管理制度」。

核心机制：

  • JSON 物理锁：功能清单用 JSON 结构，Agent 只能改状态字段，不能删改描述
  • 三步唤醒仪式：每个 Session 开头强制执行 pwd → git log → progress.txt
  • Git 存档与回滚：代码改动全部 Git 存档，陷入死胡同时 git revert 回滚
  • Context Reset：上下文撑爆时，直接换新 Agent，只给交接单

第二层：终结无政府状态，走向并发与效率

核心问题：多个 Agent 同时工作时的连环车祸。

Cursor 的崩塌：几百个 Agent 共享项目，20 个同时工作时吞吐量降到 2-3 个 Agent 水平，互相等待，集体摸鱼。

解决方案：

  • Cursor 的三层阶级：Planner（规划器）→ Worker（执行器）→ Judge（裁判）
  • DAG 引擎单行道：Planner 没吐排期表，Worker 被锁死
  • 状态机门控：长时间任务启动前必须提交完整计划、等待批准

Anthropic 的并行编译器实验：

  • 16 个 Claude 实例并行写 C 编译器
  • 引入 GCC 作为标准答案参照：用「二分查找」定位 Bug，几轮下来精确定位问题文件
  • 近 2000 个 Session，两周时间，两万美金 API 费用，产出 10 万行编译器

关键机制：大规模并发控制，用物理状态机压制个体盲目性。

第三层：戳破盲目自信

核心问题：Agent 评估自己工作时「自信地赞美」，哪怕质量平庸。

Anthropic 的对抗机制：

  • Generator-Evaluator 对抗：做事的和评判的分开，灵感来自 GAN
  • 校准评估者：让 Evaluator 保持怀疑态度，亲自动手验货
  • Sprint Contract：Generator 和 Evaluator 先协商「做完长什么样」，像甲方和施工队签验收标准

Cursor 的 8 通道并行盲审：

  • 同一个代码差异，拉起 8 个独立 Bugbot
  • 代码差异打乱顺序，各自独立发现 Bug
  • 多数投票合并，只留真信号

沙盒隔离的必需性：

  • Agent 会篡改测试环境本身（把 assert x == 5 改成 assert True）
  • 测试环境必须锁定为只读状态，考生碰不到试卷和评分标准

做加法之后学会做减法

Anthropic 的拆除实验

随着 Opus 4.5/4.6 登场，Anthropic 开始拆自己造的东西：

  • Context Reset：拆了，新模型上下文管理能力强到不再需要
  • Sprint Contract：拆了，新模型能自己把控节奏
  • Evaluator：从每轮对抗改为最后一轮做 QA

核心认知：Harness 的每个组件都编码了一条「模型做不到什么」的假设。当假设不再成立，组件就该走了。

真正的护城河

不是 Harness 的厚度，是追踪补偿面迁移的速度。

  • 模型每强一分，Harness 重心就移一寸
  • 能拆，说明之前搭得有效；拆得快，说明一直知道在补偿什么
  • 最轻量的干预（调 Prompt）反而最有效，但前提是 Harness 已经搭好

正在发生的「补偿面迁移」

账外发现：三个新维度

源码里出现的新系统，不在「执行长程任务」范畴，而是「好用、可控、可商业化」：

1. KAIROS（常驻守护程序）

  • 不等你开口，系统定时拍它肩膀问「现在需要你做什么吗」
  • 硬性限制：打断用户超过 15 秒的操作自动延后
  • Agent 从被动工具变成主动助手

2. YOLO Classifier（风险分类器）

  • 每个操作打风险标签：读文件直接放行、写文件分快速通道/完整审批、执行命令永远审批
  • 三种判定：放行、软拒绝、硬拒绝
  • 会学习：连续拒绝某类操作几次后，以后直接阻断

3. Hooks（钩子）

  • 流水线 8 个关键节点埋插槽，任何人可塞检查脚本
  • 壳从封闭产品变成开放平台

共同特征：不是执行长程任务必须的，但对效率、自定义、商业防御是必须的。

补偿面不只是在迁移，它在膨胀——从 Harness 向 Infra 蔓延。
