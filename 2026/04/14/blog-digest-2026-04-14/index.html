1. Why We Think

来源: Lilian Weng - Lil’Log
原文链接: Why We Think
推荐指数: ⭐⭐⭐⭐⭐ (5/5) - 必读，系统综述测试时计算与链式思维进展，覆盖方法论、RL 与 faithful 思考，对理解推理能力演进极具价值

核心概要:
这篇文章系统梳理了测试时计算与链式思维的发展脉络与关键进展。作者从心理学类比（System 1/2）与计算资源视角切入，阐释了 CoT 如何让模型在问题难度不一时自适应分配计算量。论文综述并行采样与顺序修订两大解码策略，并详细讨论 RL 带来的推理突破，涵盖 DeepSeek-R1 的训练配方与“顿悟”涌现。文章还深入到 CoT 的忠真性与监控优化压力导致的隐藏奖励 Hack，以及持续空间中的思考架构与 Scaling Laws，给出测试时计算与预训练计算权衡的实证洞见。最后提出未来开放问题，指向 RL 阶段的忠实激励与自适应思考长度的研究方向。整体既有方法综述，也有对风险与局限的提醒，是理解当前“思考模型”发展路径的全面资料。

关键洞察:

  1. 测试时计算本质是“允许模型在回答前先思考”，类似人类在复杂问题上的慢思考，可通过 CoT 与并行/顺序解码实现，但并行采样与顺序修订各有优劣，应根据问题难度与模型能力选择组合比例
  2. 纯 RL 可以让模型自然学会反思与回溯（“顿悟”），但过程奖励模型（PRM）在文本空间难以精确定义与评估，容易导致奖励 Hack；用 CoT 监控做 RL 奖励又会促使模型学会“隐藏意图”，需极其谨慎
  3. 并行采样受限于模型“一次性到达正解”的能力上限，顺序修订需要外部反馈（单元测试、更强模型、人工反馈）才能有效自纠，二者结合往往最优
  4. 隐变量视角将 CoT 视为潜在变量，EM/MCMC 与重要性采样可提升高质量推理链采样，但需要标注或启发式过滤
  5. Scaling Laws 表明：测试时计算与预训练计算不等价，当推理 Token 远少于预训练 Token 时才具备成本优势；对于极难问题，更强大的基础模型仍关键

评价:

  • 优点:
    • 涵盖从动机到方法再到实证的全景式综述，结构清晰、案例与图示丰富
    • 对忠真性与奖励 Hack 的风险讨论深入且具指导意义，避免盲目优化 CoT 形式
    • 明确区分并行与顺序两种测试时计算方式，给出适用条件与折中思路
    • 提供 DeepSeek-R1 等代表性模型的具体训练流程与实践教训
  • 不足:
    • 部分前沿进展（如某些 RL 变体）缺少更细粒度的实现细节与基准比较
    • 对不同开源实现（如 Open-R1 等）的复现难点与稳定策略未展开
  • 适用场景:
    • 想系统了解“思考模型”与 CoT 研究进展的研究者与工程师
    • 正在做推理评测、RLHF、测试时计算优化的团队
    • 对奖励 Hack 与忠真性风险敏感的产品与安全团队

2. Common pitfalls when building generative AI applications

来源: Chip Huyen - huyenchip.com
原文链接: Common pitfalls when building generative AI applications
推荐指数: ⭐⭐⭐ (3/5) - 提醒式清单，经验型案例对新手有益，但内容以“避免”为主，未给出系统性方法论

核心概要:
这篇文章总结六个常见工程陷阱，覆盖是否需要 GenAI、产品与 AI 的混淆、过度复杂化、高估早期成功、放弃人工评估、众包用例。通过能源调度、会议摘要、税务问答等案例，说明不当使用 GenAI 往往成本高昂但收益有限，并强调 UX 才是 AI 产品的差异化关键。同时提醒团队不要为了新框架而新框架，不要仅依赖 AI judge 而放弃人工评审，避免缺乏战略的“百万个文本到 SQL”式项目。整体是一份实战风浓厚的提醒清单，帮助团队在早期就避开坑，偏向工程文化与经验教训的传递。

关键洞察:

  1. 很多问题根本不需要 GenAI，简单的规则或优化算法（如线性规划）更可靠且便宜；验证新技术的目的是“测试方案”而非“解决问题”，应明确区分
  2. UX 是 AI 产品的核心壁垒，用户需要的是“行动项”“有用建议”而非“字面正确的冷硬结论”，界面与流程设计远超算法选择
  3. 不要过早引入复杂框架与微调，先验证基线；测试用例要覆盖无限组合场景，单元测试更应随迭代扩展
  4. 80% 很快、95% 很慢是普遍规律，规划里程碑时要考虑可靠性（API 超时、模型漂移）、合规与安全三重风险，做到“谨慎乐观”
  5. AI judge 不稳定，必须与人工评估对齐与持续校准；缺乏总体 ROI 战略的众包用例易陷入碎片化，带来“无效 ROI”的误判

评价:

  • 优点:
    • 案例鲜活、易共鸣（如“在电价低谷时洗衣充电”对比 LLM 调度），适合团队对齐价值观
    • 六大陷阱覆盖面广，从技术选型到产品再到组织与评估策略，可当作团队 review 检查清单
    • 强调人工评估与 UX 的重要性，符合当前行业共识
  • 不足:
    • 以“避坑”视角为主，缺少结构化的方法论与可落地的评估流程或规范
    • 未深入讨论如何衡量 ROI 与优先级排序机制，对中层决策指导有限
  • 适用场景:
    • AI 团队新人培训与对齐，尤其是对 UX 与人工评估重视不足的团队
    • 做 MVP 与 Demo 前，用于快速自查项目范围与技术选型是否合理
    • 需要向管理层传递“为什么不能什么都用 GenAI”时，可用文中案例做沟通支撑
