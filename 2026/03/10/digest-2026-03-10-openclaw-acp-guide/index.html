文摘：OpenClaw 的 ACP 协议怎么用？如何接入 Claude Code、Codex、Gemini CLI

Mar 10 2026 文摘

最近不少人第一次看到 OpenClaw 的 ACP（Agent Client Protocol）都会有点懵：它到底是个什么东西？为什么文档里说能操作 Claude Code、Codex、Gemini CLI 这些 AI 编程工具？它和 OpenClaw 自己的 sub-agent 又是什么关系？

如果只说一句话，我会这样概括：

ACP 是 OpenClaw 连接“外部 AI 编程 CLI / Harness”的统一接口层。

OpenClaw 负责理解你的需求、保存上下文、分发任务；ACP 负责把任务交给 Claude Code、Codex、Gemini CLI、OpenCode 这类外部编程代理去执行。

这意味着，OpenClaw 并不是自己“假装成” Claude Code，而是把 Claude Code、Codex 这类工具接进它的运行体系里，让它们变成一个个可调度、可追踪、可持续对话的执行端。

ACP 到底解决了什么问题？

如果你直接使用 Claude Code 或 Codex CLI，通常会遇到几个现实限制：

  1. 上下文分裂：代码上下文、需求背景、历史决策、客户信息分散在不同地方
  2. 会话不稳定：一个任务做完之后，后续追问不一定还能延续原有上下文
  3. 调度能力弱：缺少统一的生命周期管理，例如创建、取消、重新绑定、状态查看
  4. 多代理难编排：很难把“会理解业务的调度层”和“只负责写代码的执行层”分开

OpenClaw 的思路是把系统拆成两层：

  • 编排层（OpenClaw）：理解业务需求、保留长期上下文、管理会话和路由
  • 执行层（ACP Harness）：真正去读仓库、写代码、跑测试、修 Bug

于是，ACP 的角色就变得很清晰：它让外部 AI 编程代理不再是孤立工具，而是变成 OpenClaw 可以统一调度的一种“外接执行引擎”。

ACP 和 sub-agent 的区别

这是最容易混淆的一点。

OpenClaw 里常见有两种委托执行方式：

1. sub-agent

这是 OpenClaw 原生的子代理运行方式。

适合场景：

  • 普通调研
  • 资料整理
  • 写文档
  • 分工执行一些内部任务
  • 需要在 OpenClaw 自己的运行模型里完成的工作

2. ACP

这是通过 ACP backend 去驱动外部 harness 的方式。

适合场景：

  • 明确想让 Claude Code 干活
  • 明确想让 Codex 操作代码库
  • 想用 Gemini CLI 做代码分析或改动
  • 想把某个“真实编程代理 CLI”纳入 OpenClaw 工作流

所以可以粗暴地记成一句话：

要外部 AI 编程 CLI，用 ACP；要 OpenClaw 内部子代理，用 sub-agent。

文档中对应关系也非常直接：

  • sub-agent：sessions_spawn 默认运行方式
  • ACP：必须显式指定 runtime: "acp"

在 OpenClaw 里，ACP 是怎么触发的？

主要有两种方式。

方式一：自然语言触发

你可以直接对 OpenClaw 说：

  • “用 Codex 跑这个任务”
  • “开一个 Claude Code 会话处理这个仓库”
  • “用 Gemini CLI 看一下这个 repo 的测试失败原因”

理论上，OpenClaw 应该自动识别这是一类 ACP harness 请求，然后把任务路由到 ACP runtime，而不是 sub-agent runtime。

方式二：显式用 /acp 命令

如果你想要更强的控制感，可以直接用 ACP 指令。

  • /acp steer
  • /acp cancel
  • /acp close
  • /acp doctor
  • /acp permissions

为什么它能操作 Claude Code / Codex / Gemini CLI？

因为在 OpenClaw 的架构里，这些工具被抽象成了 ACP 的“目标 harness”。

换句话说，OpenClaw 并不关心底层是 Claude Code 还是 Codex，它关心的是：

  • 目标 agent 是谁（agentId）
  • 会话是否要持续
  • 是否要绑定到某个 thread/topic
  • 这次任务是否取消、继续、改模型、改权限

因此，Claude Code、Codex、Gemini CLI 这些原本各自独立的工具，被纳入了一个统一管理面：

  • 会话创建
  • 会话状态查询
  • 会话取消
  • 会话关闭
  • 会话持续上下文
  • 输出回传到当前聊天界面

这也是为什么 ACP 的价值并不只是“多支持几个 CLI”，而是把这些 CLI 纳入了 OpenClaw 的统一调度框架。

最常见的两种使用模式

一次性任务（oneshot / run）

适合：

  • “分析这个报错”
  • “总结这个仓库的 failing tests”
  • “找出这个 PR 的潜在风险”
  • “给出修复思路，但不用一直保留上下文”

特点：

  • 跑完就结束
  • 像临时叫来一个外部编程代理做事
  • 成本和管理复杂度较低

持续会话（persistent / session）

适合：

  • 持续维护同一个仓库
  • 围绕同一任务反复追问
  • 在一个 thread 里长期和某个 coding agent 协作
  • 多轮修 bug、改代码、跑测试、继续修

特点：

  • 上下文连续
  • 更像在聊天界面里长期挂着一个 Claude Code / Codex 工作线程
  • 适合真正进入“AI 编程协作”状态

ACP 最大的工程坑：权限

这一点非常关键。

文档明确指出：ACP 会话是非交互式的。也就是说，它没有本地终端里那种 TTY 弹窗让你批准“是否允许写文件”“是否允许执行 shell”。

这会导致一个常见问题：

Claude Code / Codex 需要写文件或执行命令时，如果权限策略不匹配，ACP 会话可能直接失败。

acpx 相关权限主要有两个配置。

permissionMode

控制哪些操作可以不经提示直接通过：

  • approve-all：自动批准写文件和 shell 命令
  • approve-reads：只自动批准读操作；写和 exec 会触发权限问题
  • deny-all：全部拒绝

nonInteractivePermissions

控制当本来应该弹出权限确认，但 ACP 又没有交互式终端时怎么办：

  • fail：直接报错终止
  • deny：静默拒绝，尽量继续执行

文档特别提醒：OpenClaw 当前默认常常是：

  • permissionMode=approve-reads
  • nonInteractivePermissions=fail

另一个关键点：ACP 跑在 host，不在 sandbox

文档里还有个容易被忽略但很重要的限制：

ACP 运行在宿主机（host runtime），而不是 OpenClaw sandbox 内部。

这会带来两个直接后果：

  1. 如果当前会话本身是 sandboxed，ACP spawn 可能被阻止
  2. 如果你要求的是强沙箱隔离，就不该选 ACP，而应该使用 sub-agent

这也解释了 ACP 的本质：它更像是“通过 OpenClaw 调用宿主机上的真实外部编程工具”，而不是 OpenClaw 内部的纯抽象代理。

线程绑定：为什么它特别适合 Discord / Telegram 这类场景？

ACP 的一个很强的能力是thread-bound session。

文档中说明，在支持 thread/topic binding 的渠道里，OpenClaw 可以把某个 ACP 会话绑定到一个 thread 或 topic 上。之后：

  • 这个 thread 里的后续消息继续路由给同一个 ACP 会话
  • 输出持续回到同一个 thread
  • 直到会话被关闭、解绑、过期或超时

这种设计非常适合真实工作流：

  • 一个 thread 对应一个仓库问题
  • 一个 topic 对应一个长期项目
  • 一个 coding agent 在这个上下文中持续干活

它让聊天界面真正变成一个“可持续协作的开发操作台”，而不是一问一答的临时对话框。

一个更准确的定位：OpenClaw 是编排层，ACP 是外部执行层

很多人第一次接触 ACP，容易把它误解成“OpenClaw 的另一个工具”。其实更准确的理解应该是：

  • OpenClaw = 编排层 / 聊天入口 / 上下文管理层
  • ACP = 外部 AI 编程代理的统一接入和控制层

于是，当你说：

  • “用 Claude Code 看看这个仓库”
  • “用 Codex 修一下这个 bug”
  • “用 Gemini CLI 做代码审查”

本质上是让 OpenClaw 做五件事：

  1. 识别这是 ACP 请求
  2. 选择目标 harness（如 claude / codex / gemini）
  3. 创建 ACP session
  4. 把任务发送给外部 coding agent
  5. 将结果和状态回传到当前聊天上下文

结语

从系统设计角度看，ACP 的意义并不是“OpenClaw 又支持了几个模型”，而是它为 AI 编程 CLI 的工程化接入 提供了一套统一控制面。

它把原本散落的 Claude Code、Codex、Gemini CLI 这类工具，纳入了一个更完整的工作流框架：

  • 有统一入口
  • 有统一生命周期
  • 有统一会话管理
  • 有线程/话题绑定
  • 有权限策略
  • 有状态控制与诊断能力

如果说 OpenClaw 解决的是“如何在聊天界面中持续组织复杂工作”，那么 ACP 解决的就是：

如何把真正能干代码活的外部 AI 代理，稳定地接进这个系统里。

对于想搭建 AI-Native 开发工作流的人来说，ACP 不是一个边缘功能，反而是 OpenClaw 走向“可操作 Claude Code / Codex / Gemini CLI”的关键桥梁。
