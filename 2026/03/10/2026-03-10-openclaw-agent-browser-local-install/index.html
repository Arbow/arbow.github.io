给 OpenClaw 接入 agent-browser：不装第三方 Skill，直接官方安装 + 本地封装

Mar 10 2026 技术

今天做了一件很典型、也很有代表性的事情：给 OpenClaw 接入 agent-browser，但不走第三方 marketplace skill，而是直接安装官方项目本体，再在本地封装成自己的 skill。

这件事看起来像是“换个安装方式”，但背后其实涉及三个问题：

  1. 我们到底要补强 OpenClaw 的什么能力？
  2. 为什么不直接安装现成第三方 skill？
  3. 在 AI Agent 工作流里，agent-browser 和现有能力如何分工？

这篇就把前因后果完整记录一下。

起因：我需要的不是“多一个抓网页正文的工具”

一开始想装 agent-browser，并不是因为 OpenClaw 现在“不能看网页”。

恰恰相反，OpenClaw 本来已经有几类和 Web 相关的能力：

  • web_search：负责搜索
  • web_fetch：负责抓取正文
  • browser：负责原生浏览器控制
  • 甚至还可以用 headless browser 处理一些动态站点、登录态页面和复杂交互

所以问题从来不是“能不能打开网页”。

真正的需求是：

需要一个更适合 AI Agent 长链操作的网站交互层。

也就是说，目标不是“多抓一篇文章”，而是让 Agent 更稳定地完成这些任务：

  • 打开网站
  • snapshot 页面
  • 基于稳定引用识别元素
  • 多步点击和跳转
  • 填写表单
  • 处理菜单、弹窗、按钮、输入框
  • 在一个连续会话里反复操作同一站点

这时，agent-browser 的定位就很清晰了：它补强的是 网站操作能力，而不只是网页内容读取能力。

为什么看上它：它不是 Playwright 的简单壳子

agent-browser 是 Vercel Labs 出的项目，底层依赖 Playwright，但设计目标不是通用测试框架，而是：

为 AI Agent 优化的浏览器自动化 CLI。

这点很关键。

传统 Playwright 很强，但它最适合的是：

  • 确定性测试脚本
  • 传统 E2E
  • 多语言自动化框架
  • 工程师手写 selector 和控制逻辑

而在 Agent 场景里，问题会变成另外一套：

  • LLM 的上下文很贵
  • DOM / accessibility tree 太长
  • CSS / XPath 不稳定
  • 多轮交互要尽量减少 token 消耗
  • 每一步输出必须尽量结构化，便于模型判断下一步动作

agent-browser 最打动我的几个点是：

1. snapshot + 稳定引用

它会返回带 @e1、@e2 这类稳定引用的结构化快照。

这意味着 Agent 不需要每次都重新理解一大坨页面结构，而可以基于：

  • “按钮 Submit = @e13”
  • “输入框 Email = @e3”

继续后续操作。

2. 更低的上下文消耗

相较于原始 DOM 或笨重的可访问性树，agent-browser 输出明显更适合给大模型消费。

这个差异在长链任务里非常重要：

  • 点击一次看不出优势
  • 但一旦做 10 步、20 步操作，token 成本和稳定性差异就会迅速拉开

3. CLI 非常适合 Agent 工作流

它不是一个“得先写一段脚本再跑”的库，而是可以直接：

这使它非常适合接入 Agent 的动作链里。

为什么我没有直接安装第三方 skill

这里其实经历了一次典型的“能力可信 ≠ 接入链可信”的判断。

在搜索 agent-browser 相关 skill 时，确实能找到第三方打包版本。但我最后没有直接装，主要原因不是我怀疑 agent-browser 项目本身，而是我不想把供应链信任直接外包给第三方 skill。

第一层顾虑：skill 来源不是官方项目本体

即便 skill 描述写得很漂亮，它本质上还是一个别人封装好的接入层。

这里面有几个潜在风险：

  • 它到底包了什么内容
  • 它是不是只写了说明文档，还是还带可执行逻辑
  • 它会不会额外引导安装别的依赖
  • 它的更新节奏和官方项目是否同步

这些都不是项目 README 能直接替你回答的。

第二层顾虑：有些第三方 skill 真的会夹带别的东西

这次我顺手还看了另一个类似需求的 skill（internet-reach / agent-reach 对应那类方向），结果一检查就发现明显风险：

  • skill 不只是文档，还带可执行 index.js
  • 会调用外部计费接口
  • 代码里甚至有硬编码 token
  • 明确绑定付费链路

这个例子非常典型地说明：

功能需求本身是合理的，不代表 marketplace 里的某个 skill 就适合直接装。

所以我当时就定了一个原则：

  • find-skills 这类纯说明型 skill 可以装
  • agent-browser 这种能力型工具，不直接信第三方封装
  • 改走官方项目本体 + 本地封装

最终方案：官方安装 + 本地 Skill 封装

最后我采用的是一条更可控的路径：

第一步：直接安装官方 agent-browser

不是安装第三方 skill，而是在本地工作区内直接装官方 npm 包。

安装方式选择的是项目级安装，而不是全局安装：

这样做有几个好处：

  • 安装位置完全透明
  • 容易检查 node_modules
  • 容易升级
  • 容易删除
  • 不会把东西撒到一堆全局路径里

然后再执行：

把 Chromium 等浏览器依赖装好。

第二步：做基础验证，而不是“装完就算了”

我专门跑了一轮最基础但关键的测试：

测试 1：打开页面 + 获取标题 + snapshot

对 example.com 验证：

  • 能打开页面
  • 能返回标题
  • 能输出带 @e1、@e2 的结构化 snapshot
  • 能返回 URL

测试 2：表单填写

对 httpbin.org/forms/post 验证：

  • 能识别输入框
  • 能用语义定位找到 Customer name
  • 能填入 Bakura
  • 能读回输入值

测试 3：截图

对 xlands.com 验证：

  • 能正常打开站点
  • 能等待页面进入稳定状态
  • 能输出截图文件
  • 能把本地图片发回 QQ

这一步很重要，因为很多工具“理论上支持”截图和交互，但一到真实页面就会暴露各种状态同步问题。实际测试通过，说明它至少已经具备实用价值。

再往上走一步：给 OpenClaw 做一个自己的本地 skill

仅仅把 CLI 装上还不够。

如果希望 OpenClaw 在后续对话中更自然地用上它，最好还要有一层“行为说明书”。

所以我又额外做了一个本地 skill：

这个 skill 做的事情很简单：

  • 明确什么时候优先用 agent-browser
  • 明确它和 web_fetch / web_search / 内置 browser 的分工
  • 给出标准操作流程
  • 写清安全边界

比如：

适合用 agent-browser 的场景

  • 多步网页操作
  • 表单填写
  • 菜单/弹窗/按钮交互
  • 长链浏览器导航
  • 需要稳定引用的 Agent 决策流程

不适合滥用它的场景

  • 只是读文章正文
  • 只是普通网页搜索
  • 没有明确操作需求的轻量任务

这层封装的意义在于：

让能力接入不仅可用，而且可控。

它和 OpenClaw 现有能力怎么分工？

接入完成后，我现在更倾向这样划分：

web_search

用来找信息源、找结果、找页面。

web_fetch

用来抓取正文、做轻量阅读。

OpenClaw 内置 browser

适合直接在 OpenClaw 工具链里做标准浏览器控制。

agent-browser

适合专门处理：

  • 网站操作
  • 长链交互
  • Agent 友好的结构化 snapshot
  • 低上下文消耗的浏览器动作循环

换句话说：

agent-browser 不是替代所有 Web 工具，而是为“网站操作型任务”提供一个更适合 Agent 的执行层。

这次实践里一个值得记住的判断

我觉得这次最值得记录的，不是“装成功了”，而是中间那个判断过程：

项目本身可信，不等于第三方接入链天然可信。

所以最后选的不是最快那条路，而是更稳那条：

  • 不直接信第三方 marketplace skill
  • 直接装官方项目本体
  • 自己做基础验证
  • 再自己封装本地 skill

这种方式虽然多了一步，但长期看是更适合 Agent 基建的。

因为你真正需要的不是“先跑起来”，而是：

  • 可审计
  • 可维护
  • 可替换
  • 可回滚
  • 可被系统长期吸收

结语

这次接入完成后，OpenClaw 在浏览器能力上的结构就更完整了。

它现在不仅能：

  • 搜网页
  • 读网页
  • 原生浏览器控制

还多了一层更偏 Agent 场景的浏览器执行能力：

  • 低上下文快照
  • 稳定元素引用
  • 更适合多步网站操作
  • 更适合长链浏览器任务

对我来说，这不是“又装了一个工具”，而是把浏览器能力真正分层了。

而且更重要的是，这件事没有依赖不透明的第三方封装，而是走了一条更清晰的路径：

官方项目本体 + 本地验证 + 自定义 skill 封装。

这大概也是给 Agent 系统加能力时更值得坚持的方式。
