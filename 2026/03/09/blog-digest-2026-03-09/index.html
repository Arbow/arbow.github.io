1. 量化代理编码评估中的基础设施噪音

Anthropic 的研究发现，代理式编码基准（如 SWE-bench 和 Terminal-Bench）的分数可能受到基础设施配置的显著影响。在内部实验中，资源配置的差异导致了 6 个百分点的差距（p < 0.01），这与排行榜上顶级模型之间的差异相当。

研究团队在 Terminal-Bench 2.0 上测试了六种资源配置，从严格执行（1x）到完全无限制。结果发现，成功率随着资源余量的增加而提升，主要原因是基础设施错误率从 5.8%（严格执行）下降到 0.5%（无限制）。

关键发现：在 3x 余量之前，额外的资源主要解决基础设施可靠性问题；超过 3x 后，额外资源开始实际帮助代理解决之前无法解决的问题。这意味着资源限制实际上可能改变评估测量的内容——严格的限制会奖励非常高效的策略，而宽松的限制则更宽容，奖励能更好利用所有资源的代理。

来源：Quantifying infrastructure noise in agentic coding evals

2. 用并行 Claude 团队构建 C 编译器

这篇文章描述了一个有趣的实验：Anthropic 研究员使用 30 个 Claude Opus 4.5 实例组成的”团队”，在不到 4 小时内构建了一个完整的 C 编译器。这个项目包括词法分析器、解析器、类型检查、代码生成和优化器，生成的汇编代码质量足以运行复杂程序（如 Python 解释器）。

核心机制类似于人类团队：每个 Claude 实例负责编译器的不同组件，通过定义良好的接口通信。一个”项目经理”实例负责协调，管理任务分配、接口定义和集成。团队使用 Git 进行版本控制，每个成员都有自己的分支。

关键挑战：

  • 接口一致性：确保所有组件使用相同的数据结构表示
  • 错误处理：定义统一的错误格式和传播机制
  • 测试策略：每个组件有单元测试，系统有集成测试

实验结果显示，编译器生成的代码在运行速度和代码大小方面都达到了生产级质量。虽然人类开发者仍然在验证代码正确性和边缘情况处理方面具有优势，但这个实验展示了代理团队在软件开发中的巨大潜力。

来源：Building a C compiler with a team of parallel Claudes

3. 设计 AI 抗性技术评估

随着 AI 能力的提升，评估技术候选者变得越来越困难。Anthropic 性能工程团队从 2024 年初开始使用一个 take-home 测试，候选人需要优化在模拟加速器上运行的代码。超过 1000 名候选人完成了这个测试，现在有数十名工程师在公司工作，包括启动 Trainium 集群和自 Claude 3 Opus 以来每个模型的工程师。

但每个新的 Claude 模型都迫使他们重新设计测试。当给定相同的时间限制时，Claude Opus 4 的表现超过了大多数人类申请者。然后 Claude Opus 4.5 甚至超过了最强的候选人。人类在无时间限制的情况下仍然可以超越模型，但在 take-home 测试的约束下，团队无法再区分顶级候选人和最强大模型的输出。

文章描述了迭代三个版本的 take-home 测试的尝试，以及每次学到的关于什么使评估对 AI 辅助具有鲁棒性，以及什么不具有。最终，团队设计了一个使用非常受限指令集的测试，类似 Zachtronics 游戏，优化解决方案以最小化指令计数。这个测试故意不提供可视化或调试工具——构建调试工具是被测试的内容的一部分。

来源：Designing AI-resistant technical evaluations

4. 为 AI 代理揭秘评估

这篇长篇文章详细介绍了如何为 AI 代理设计严格且有用的评估。评估使问题和行为变化在影响用户之前可见，并在代理的生命周期中产生复合价值。

评估的结构：一个评估是 AI 系统的测试：给 AI 输入，然后应用评分逻辑到输出来测量成功。单轮评估很简单：提示、响应和评分逻辑。但随着 AI 能力的进步，多轮评估变得越来越普遍。代理评估更加复杂，因为代理在许多轮中使用工具、修改环境中的状态并适应，这意味着错误可以传播和复合。

为什么要构建评估：

  • 评估迫使产品团队明确说明代理的成功意味着什么
  • 提供基线和回归测试：延迟、令牌使用、每任务成本和错误率可以跟踪
  • 成为产品和研究团队之间的最高带宽通信通道，定义研究人员可以优化的指标

评估代理的方法：

  • 编码代理：通常依赖明确定义的任务、稳定的测试环境和生成的代码的彻底测试
  • 对话代理：通常依赖可验证的最终状态结果和捕获任务完成和交互质量的评分
  • 研究代理：组合评分类型：基础性检查验证声明得到检索来源支持，覆盖检查定义好的答案必须包含的关键事实
  • 计算机使用代理：通过人类使用的相同界面与软件交互：截图、鼠标点击、键盘输入和滚动，而不是通过 API 或代码执行

从零到一：优秀代理评估的路线图：

  1. 从 20-50 个简单任务开始
  2. 从你已经手动测试的内容开始
  3. 编写明确任务和参考解决方案
  4. 构建平衡的问题集
  5. 构建稳健的评估装备和稳定环境
  6. 设计评分器
  7. 检查记录
  8. 监控能力评估饱和
  9. 保持评估套件的健康

文章还讨论了评估与其他方法（生产监控、A/B 测试、用户反馈、手动记录审查和系统性人类研究）如何结合，以提供对代理的全面理解。

来源：Demystifying evals for AI agents

总结

今日推荐的 4 篇文章都来自 Anthropic Engineering Blog，涵盖了 AI 代理评估的多个方面：基础设施噪音对评估的影响、代理团队在软件开发中的应用、设计对 AI 具有鲁棒性的评估，以及如何为各种类型的代理构建有效的评估系统。这些文章提供了宝贵的实践经验，对于正在构建或使用 AI 代理的开发者和研究者都具有重要参考价值。
