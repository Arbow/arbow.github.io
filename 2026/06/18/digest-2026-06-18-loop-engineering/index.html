文摘：Loop Engineering 概念解析、思考与实践

Jun 18 2026 文摘

文章概要

这篇文章来自「飞雪谈AI」（作者：飞樰），系统化地拆解了近期 AI 圈密集讨论的新概念——Loop Engineering（循环工程）。Addy Osmani 在 6 月初正式定义了这个概念，其核心一句话是：

Loop engineering is replacing yourself as the person who prompts the agent. You design the system that does it instead.

翻译过来就是：把你从”给 Agent 写提示词的人”这个位置上替换掉，转而设计一套能自动完成这件事的系统。

Agent Loop ≠ Loop Engineering

文章做出了一个关键区分：

Agent Loop 是底层基础——大模型输出 Function Call → 工具执行 → 结果回传 → 再推理，形成循环。这是 Agent 的”默认基础设施”，ReAct、Ralph Loop 等都是这个原理的变种。

Loop Engineering 则是构建在 Agent 之上的新范式。它不是关注底层 Agent 怎么转，而是关注：如何设计一个把”开发→验证→反馈→调优”完整链路串起来的外部闭环，让 Agent 自己跑起来，而不是人一直在旁边喊”继续””报错””回滚”。

从 Coding 到 Vibe Coding，是我们从”写代码”变成了”提需求”；而从 Vibe Coding 到 Loop Engineering，则是从”提一个需求”变成了”提一套闭环流程”。

Loop Engineering 六大核心框架

根据 Addy Osmani 的定义，一个完整的 Loop 包含：

组成部分 作用
Automations（自动化） 定时循环触发，如 Codex /goal、Claude Code /loop + Cron 调度
Worktrees（工作树隔离） Git Worktree 让多 Agent 并行不冲突
Skills（可进化技能包） 每次循环中自我沉淀经验，越跑越聪明
Connectors/Plugins（连接器） MCP 工具接入，让 Agent 触达真实世界
Sub Agents（子智能体） 独立验收/探索/设计分支，角色隔离打破认知盲区
State（状态管理） 追踪任务进度，可用 Markdown 或对接 Linear 等

其中 Sub Agents 的设计尤为巧妙：主 Agent 完成开发后，由独立的验收 Sub Agent 来检查——刻意制造”博弈”关系，避免”既当运动员又当裁判员”。

实践案例：文本分类

作者用一个文本分类任务做了对比：

  • 传统方式：写 Prompt → 输出结果 → 人工检查 → 手动反馈调整 → 沉淀 Skill
  • Loop 方式：把验证标准和迭代逻辑写进 Loop 定义，设定量化目标（如”100条测试数据准确率≥95%”），Agent 自主循环打磨逼近目标

关键差异在于：这个过程完全由 Agent 自主驱动，不依赖外部框架。Agent 自己跑测试、改逻辑、沉淀能力——在 AI 时代，你未必需要先造一个全新框架，用 Claude Code 或 Codex 写一个结构化 Loop 就能”跑”出很多原本需要工程化封装的能力。

Loop 不是银弹

文章最后给出了清醒的提醒：

  1. 需求必须更明确：以前模糊的需求靠人工纠偏，用了 Loop 后中间不参与，开头没写清楚就会跑偏、烧 token
  2. 验证标准要清晰：Loop 对使用者描述需求的能力要求更高了
  3. 固定流程建议沉淀为脚本/Skill，不要每天重跑 Loop——既费 token，模型实现路径也可能漂移
  4. 如果发现自己很难把控 Loop 效果，老老实实回到 HITL 模式可能更经济

技术方法论没有对错，只有适不适合。

我的思考

这篇文章把 Loop Engineering 的概念讲得很清晰，尤其是”Agent Loop vs Loop Engineering”的区分点出了很多人混淆的地方。六大核心框架的梳理也很有参考价值——其中 Automations、Worktrees、Sub Agents、Skills 这几个维度在 Hermes Agent 的实践中都有直接对应（Cron 定时任务、隔离子代理、Skill 进化、子任务编排）。

不过文中对”Loop 就是循环起来的 Pipeline”的类比可能让一些读者觉得概念过于宽泛。实际上 Loop Engineering 的核心增量不在于 Pipeline 本身（CI/CD 早已是成熟实践），而在于让 Agent 自主驱动验证和迭代——也就是把 HITL 中人的角色从”循环参与者”降级为”目标定义者和最终确认者”。这个区分是真正的范式变化。
