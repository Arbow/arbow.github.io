文摘：Setting Up and Using the Pi Coding Agent

Jun 13 2026 文摘

背景

DeepakNess 同时是 Cursor 和 OpenAI Codex 的付费用户，但在两周前接触到了 Pi —— 一个由 Mario Zechner 创建的开源终端编程 agent。他尝试之后将 Pi 定位为副项目和实验的专用工具，写了这篇实战分享。

Pi 是什么

Pi 是一个开源终端编程 agent，核心理念是 小核心 + 可扩展。默认只提供四个工具：read、write、edit、bash。没有内置的 plan mode、sub-agents、MCP 或权限弹窗——这些都通过 extensions、skills 和 packages 按需添加。

Pi 支持多种 AI 提供商：Anthropic、OpenAI、DeepSeek、Google Gemini、Groq、xAI、OpenRouter、Kimi 等，可以用 API key 或已有订阅登录。

作者的使用策略

不同工具分工：

  • OpenAI Codex — 主项目的复杂任务
  • Cursor — 同一项目的日常轻量编码
  • Pi — 副项目、实验和一次性任务，搭配 DeepSeek、Kimi 等便宜模型

亮点案例：用 Pi + DeepSeek v4 Flash 抓取了 285K 个 URL，运行约 1.5 小时，仅花费 $1。

关键配置

模型选择

模型 用途
DeepSeek v4 Pro 默认主力，质量与成本均衡
DeepSeek v4 Flash 轻量重复任务，快且便宜
Kimi K2.6 / K2.7 Code 备选模型，限速时切换
Cursor (via pi-cursor-sdk) 复用已有 Cursor 订阅
OpenAI Codex (GPT-5.5) 需要强推理时偶尔使用

Ctrl+L 打开模型选择器，Ctrl+P 循环切换，Shift+Tab 切换 thinking 等级。作者默认用 xhigh thinking，轻量任务切 Flash。

安装的 5 个核心 Package

  1. pi-web-access — 赋予 Pi 网页搜索、页面抓取、YouTube 转录和 GitHub 仓库探索能力，开箱即用 Exa 搜索（无需 API key）
  2. pi-codex-goal — 长任务目标追踪，提供 get_goal/create_goal/update_goal 三个工具
  3. pi-vision-proxy — DeepSeek 没有视觉能力时，将图片代理转发给有视觉的模型（如 Kimi K2.6），支持感知哈希去重和批量处理
  4. pi-agent-browser-native — 在终端内提供真实浏览器：截图、点击、填表、QA 检查、页面内容提取
  5. pi-cursor-sdk — 复用 Cursor 订阅，让 cursor/auto 作为模型选项出现在 Pi 中

项目指令：AGENTS.md + APPEND_SYSTEM.md

  • AGENTS.md — 项目上下文，从 ~/.pi/agent/ 向上查找到当前目录，拼接注入 system prompt
  • APPEND_SYSTEM.md — 全局行为规则，直接追加到 system prompt，权威性高于 AGENTS.md

作者的 APPEND_SYSTEM.md 规则包括：先读本地文件再搜索网络、破坏性操作先确认、避免 AI 套话文风、使用 en dash 等。

自定义扩展

作者让 Pi 自己写了一个最小化 footer 扩展（~/.pi/agent/extensions/minimal-footer.ts），只显示目录、git 分支、模型、thinking 等级和 context 用量，放在 gist 公开分享。

作者喜欢的点

  • 快且简洁：纯终端界面，没有 Electron 和 UI 卡顿
  • 高度可定制：自己想要的 footer、package 和配置都可以自由组合
  • 多提供商支持：DeepSeek、Kimi、Cursor 之间一个快捷键切换
  • Session Tree：对话可分支、可回溯，这是其他工具没有的体验
  • 开源 (MIT)：不受任何公司功能或定价的限制
  • Compaction 好用：上下文压缩自动进行，比其他工具更快更好

我的评论

这篇文章的价值不在于 Pi 本身的介绍——Pi 的设计理念（极简核心 + 可扩展）在 coding agent 领域并不算新颖，Claude Code、Aider 等都有类似的模块化思路。真正值得关注的是作者的 工具组合策略 和 成本意识。

多工具分工而非单一替代。作者没有用 Pi 替换 Cursor 或 Codex，而是让各工具发挥所长：Codex 做重活、Cursor 做日常、Pi 做实验。这种务实的态度比”某工具能完全取代一切”的叙事靠谱得多。

低成本批量的可行性。285K URL 抓取花了 $1，这组数字很有说服力。它说明对于确定性强、可并发的批量任务，用便宜模型 + 好 prompt engineering + 长任务追踪（pi-codex-goal），可以做到极低的单位成本。这对需要大规模数据处理的团队有直接参考价值。

APPEND_SYSTEM.md 的设计思路值得借鉴。把”先读本地文件”、”破坏性操作先确认”、”避免 AI 套话”这类跨项目通用规则与项目特定的 AGENTS.md 分离，并且赋予更高优先级——这是一个很好的 prompt engineering 实践，适用于所有支持分层 system prompt 的 agent 系统。

Pi 的 Package 生态是最大看点。pi-vision-proxy 用感知哈希去重截图、pi-cursor-sdk 复用订阅……这些 package 展示了”小核心 + 社区扩展”模式的潜力。不过生态规模和成熟度仍然需要观察——文章只涉及了 5 个 package，且其中两个来自同一个作者 (Mitch Fultz)。
