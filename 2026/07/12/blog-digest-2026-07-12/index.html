一文看懂ChatGPT、Codex、Work 的差别

来源：宝玉的博客
链接：https://baoyu.io/blog/chatgpt-work-codex-guide
发布日期：2026-07-11
类型：产品分析 / 指南

核心概要

OpenAI 近期将产品矩阵重组为 Chat、Work、Codex 三个模式，宝玉这篇文章用一句话点透了三者的定位差异：Chat 回答问题，Work 帮你干活，Codex 帮你写代码。 文章从使用场景、读取内容、交付结果三个维度做了清晰的表格对比，然后深入到 Work 的实际能力——它不只是”帮我写个报告”，而是能接入 Slack、Gmail、Google Drive、Teams 等业务系统，跨应用拉取数据、整合信息，最终交付可直接使用的文档、表格或幻灯片成品。

文章还花了大篇幅讨论一个很容易被忽略但极为关键的问题：额度到底怎么算？ 答案是 Chat 的消息限额独立计算，而 Work 和 Codex 共享同一个”智能体额度池”（agentic usage）。这意味着白天用 Work 跑复杂任务，晚上用 Codex 写代码时可能会发现额度被挤占。这对同时依赖两种模式的重度用户来说是个重要的成本考量。

后半部分是本文最有价值的增量信息——ChatGPT Work 与 Claude Cowork 的横向对比。两者都是桌面端智能体，都能操作本地文件、使用 Computer Use，但在执行架构（沙箱实现）、应用连接方式（plugins vs MCP）、跨设备同步能力上有显著差异。特别是跨设备同步这一点，Claude Cowork 走得更靠前：同一会话可以在桌面、网页和手机之间流转，而 ChatGPT Work 的云端与桌面端目前仍然割裂。

关键洞察

  • 三模式定位清晰但有重叠风险：Chat/Work/Codex 的划分本质上是”一套底层 Agent 能力，两个名字（Work 和 Codex），服务于办公和编程两种场景”。OpenAI 用 Work 这个新名字把 Codex 的非编程用途正式化，既不破坏开发者对 Codex 的认知，又能吸引办公人群。
  • 额度共享是隐形成本陷阱：Work 和 Codex 共享 agentic usage 池子，用户需要意识到这两个模式会互相挤占额度。按 Token 消耗计费后，强模型 + Fast 模式会更费额度。
  • Scheduled Tasks 让 Work 具备自动化能力：支持定时任务、事件触发和持续监控，可以把”每天检查 Slack 和邮件整理简报””收到反馈自动归类”这类工作流自动化，这是从”对话工具”向”工作流引擎”的关键一步。
  • Cowork 在跨设备同步上暂时领先：Anthropic 的 Cowork 任务可以远程运行在服务器上，关掉电脑也能继续，手机上可查看进度。ChatGPT Work 的云端与桌面端不互通，这是 OpenAI 明确的 launch 限制。
  • GPT-5.6 三档型号影响成本：Sol（最强）/ Terra（均衡）/ Luna（最快最便宜），Free 和 Go 用户只能用 Terra，选择模型和 effort 级别直接影响额度消耗速度。

评价

优点：

  • 结构极其清晰，用”一句话定位 + 对比表格 + 实际例子 + 横向竞品对比”的递进结构，把一个容易混淆的话题讲得非常透彻
  • 额度共享机制、GPT-5.6 子型号定价、各订阅计划差异等”细节深坑”都覆盖到了，实用价值高
  • 与 Claude Cowork 的对比是独有增量，不是单纯转述 OpenAI 官方信息，而是给出了有判断的产品分析

不足：

  • 文章基于 X/Twitter 两条 thread 整理，部分内容（如企业案例 Zapier、Virgin Atlantic）来自 OpenAI 官方宣传，实际落地效果有待验证
  • Work 的 plugins 生态目前 60+ 连接器，但文章未评估这些连接器的成熟度和可靠性
  • 对 GPT-5.6 三个子型号的区分建议偏定性，缺乏具体场景下的实测对比

适用场景：对 OpenAI 产品矩阵感到困惑的用户、需要评估 ChatGPT Work 是否适合团队使用的决策者、关注 AI 办公平台竞争格局的产品经理和开发者。
