文摘：uv 创始人反思 AI 编程——代码审查、信任崩塌与工程师困境

Jul 05 2026 文摘

背景

Charlie Marsh，Ruff 和 uv 的创建者、Astral 公司创始人。他用 Rust 打造了比传统 Python 工具快一到两个数量级的 linter 和包管理器，GitHub 星标数以万计增长，最终在今年三月将整家公司卖给了 OpenAI。

日前，他在播客中与主持人 Ryan Peterman 深入探讨了 AI Agent 如何重塑软件工程的每一个环节。这不是一个”技术天才征服世界”的成功学叙事，而是一线实践者对当下软件开发困惑最诚实的思考：当提交 PR 的成本降到零、审查成本却居高不下，开源生态将何去何从？当写代码的乐趣被”指挥 Agent + 逐行审查”取代，编程还值得热爱吗？

核心观点

1. Python 工具链的”Rust 革命”

Charlie 在计算生物学公司工作时，接触了 Go 和 Rust，发现 JavaScript 生态早已用原生语言重写工具链（ESBuild、SWC、Bun、Deno），而 Python 生态仍停留在用 Python 写工具的阶段，性能天花板明显。Ruff 的起点很简单：为什么 Python 不能拥有同样快的工具？ 他用 9 天做出了原型，验证了这个假设。

关键洞察：linter 是验证想法的完美形态——核心逻辑简单，规则可无限扩展，用户从小核心就能立刻获得价值。相比之下，完成了 75% 的类型检查器几乎毫无用处。能够迭代式地交付、同时保持对用户有用，是建立工具势能的核心能力。

2. 开发者营销的”10 秒法则”

Charlie 对开发者营销有独到见解。他认为工程师圈子里”营销”几乎是脏话，但大量优秀项目正因为不会营销而永远无人发现。他的方法论：

  • 10 秒内抓住注意力：博客、README、宣传材料，核心问题是”当一个人打开页面，我只有 10 秒让他明白这对他有什么用”
  • 一张好的 benchmark 图是无价的：Ruff 的性能对比图在 Twitter 上被疯狂转发，不需要任何文字解释
  • 一天内响应-修复-发布的循环：每完成一次就赢得一个支持者，项目势能增加一分
  • 他研究过 OpenAI 的 DALL-E 博客，即使一个字都不读，光看图片和标题也能理解他们做了什么

3. 为什么选 Rust——从跟风到真香

坦白说，Charlie 当初选 Rust 很大程度是因为跟风（hype）。但回头看，这是一个极其正确的决定：

  • Cargo 工具链是被低估的优势：git clone 然后 cargo run/build/test 就能跑起来，这在 C/C++ 生态几乎不可想象
  • 内存安全从不在乎到真香：他开始做 Ruff 时甚至不知道内存安全是什么，现在认为这是构建可靠软件的基础
  • 他明确表示不理解为什么新项目还要用 C/C++，除非有非常特定的技术原因

4. AI 时代的代码重写——“我一行代码都没读就发布了”

这是整篇访谈最有争议的部分。Charlie 去年圣诞假期才开始用 Agent 编程，到现在几乎所有修改都通过 Codex 完成。他分享了一个极端案例：

上周我写了个个人用的自定义 Rust linter，全程由 GPT-5.5 完成，我一行代码都没读。因为那是只有我们内部使用的工具，很容易判断正确性。

但他对大规模重写持强烈保留态度，给出三层反对理由：

  1. 代码理解问题：如果代码被完全重写，人类可能就不再理解任何代码了
  2. 已知问题换未知问题：即使最好的测试套件也只是程序正确性的近似逼近。希拉姆定律（Hyrum’s Law）告诉我们，任何实现细节最终都会变成某些人依赖的行为。测试全过不代表行为一致，最终坑的是用户
  3. 责任问题：uv 被数百万工程师依赖，任何变更必须极其谨慎；而个人工具可以 YOLO 式发布。关键在于你对用户负多大责任

他观察到 Bun 仓库的运作方式与 Astral 完全不同——头号贡献者是一个 bot，人类提交的 PR 评论也明显是 agent 写的。他对此持欣赏但保留态度。

5. 对抗 AI Slop 的具体策略

Charlie 分享了团队应对 AI 生成低质量贡献的策略：

开源维护层面：

  • 制定 AI 政策，过滤净负面贡献——核心要求是”你提交的东西，你自己得理解”
  • 识别 agent 写的回复的特征：过于详尽、格式过于规范、用词过于学术、塞满各种链接、在不必要的地方投入远超人类的”努力”
  • 引用 Zig 项目提出的”贡献者扑克”概念：传统模式下，新贡献者通过反馈循环成长；agent PR 打破了这个循环，没有累积，没有成长

根本困境：

提交一个”看起来合理”的 PR 的成本已经降到了零，但审查和验证它的成本依然很高，甚至更高。以类型检查器 TY 为例，有人可能花 2 分钟写一个 PR，但团队需要花 1 小时去理解它。这种不平衡正在严重破坏开源社区生态。

团队内部层面：

  • 大量自动化验证：每个 PR 在 Valgrind 和 CodSpeed 上跑 benchmark，对比生态项目诊断信息差异
  • 提 PR 前用 Codex Review 预审多轮
  • 持续优化 agents.md 文件，把团队和 Agent 常犯的错误编码化
  • 底线标准：你必须理解 PR 里的每一行代码

6. 信任的瓦解与重建

Charlie 坦陈团队内部出现的信任变化：

以前同事提 PR，我基本扫一眼就过了，因为信任他的工作。但现在他提 PR，我得逐行细审，因为那些代码不是他写的，是 Agent 写的。

他自己也经历同样的困境——有时候第二天早上看自己昨晚提的 PR，心里想的是”等等，这代码写得也太烂了吧”。代码看起来能用、可接受，但根本达不到以前的质量门槛，这种灰色地带是最难识别的 AI slop。

7. 初级工程师的困境

Charlie 对初级工程师在 AI 时代的处境直言不讳：

我们倾向于招资深工程师，因为优秀的工程师用 AI 更猛。如果我是初期工程师，太容易掉进使用 Agent 时发生的那些坏事情里了。

他担心初级工程师缺乏”给 AI 当教练”的能力——很多时候是他在指导 Codex、纠正它、为它提供安全护栏。没有扎实基础的人，只能被 AI 带沟里去。他引用 Mitchell Hashimoto（Ghostty 作者）的实验：手写渲染器比 LLM 优化后的版本还快 100 倍，说明不先用大脑从第一性原理思考，就会满足于次优结果。

8. 从困惑到和解

访谈最动人的部分是 Charlie 的心路转变。他一度恐惧 AI 会让编程变得无趣：

我一直有个恐惧：AI 会让我们更高效，但编程会变得没那么有趣，因为我就是热爱编程本身。以后我得把所有时间花在审查代码和提示这个总犯错、但最终确实更高效的蠢 Agent 上。

但几个月后，他发现与 Agent 协作解锁了新的可能性——运行实验的成本变得极低，那些以前很难、成本很高才能回答的问题，现在几乎可以瞬间得到答案。他从构建软件中获得的价值并未消失：仔细思考数据结构布局、合并一个关闭用户问题的 PR，这些满足感依然存在。

个人评论

这篇文章值得反复读。几个让我印象深刻的点：

关于审查成本的不对称——这可能是当下开源生态最尖锐的问题。当生成 PR 的边际成本趋近于零，而验证成本不变甚至上升（因为需要甄别 AI slop），开源维护者的负担会指数级增长。Charlie 提到有人花 2 分钟生成 PR，维护者需要 1 小时审查，这种 1:30 的不对称如果不解决，依赖志愿者的开源模式将面临严峻考验。

“我一行代码都没读就发布了”——这句话在社交媒体上会很抓眼球，但 Charlie 的完整语境很重要：那是内部工具，容易验证正确性。他对 uv 这种被数百万人依赖的项目则极其谨慎。这说明真正值得关注的不是”敢不敢不看代码”，而是如何根据项目风险等级建立分层审查策略。很多 AI 编程讨论的非黑即白（要么全盘拥抱要么全面抵制）其实偏离了重点。

Mitchell Hashimoto 的 100x vs 10x 实验值得每个用 AI 写代码的人深思。LLM 擅长在现有架构内做局部优化，但不擅长从第一性原理重新思考。如果你没有能力判断”这东西理论上应该有多快”，你就会把次优当成最优交付出去。这恰恰是初级工程师最危险的陷阱——他们既缺乏判断力，又被 AI 的”看起来很厉害”所迷惑。

关于开发者营销——Charlie 对”10 秒法则”的阐述值得所有做开源的人学习。注意力经济是现实，与其抱怨用户不愿意深入阅读，不如想想如何用诚实的方式在几秒钟内传达价值。一张精心设计的 benchmark 图胜过千言万语。
