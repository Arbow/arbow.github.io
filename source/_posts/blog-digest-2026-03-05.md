### ⭐⭐⭐⭐⭐ Star Trek : Captain's Chair 初体验
**作者**: 云风 | **日期**: 2026-03-01 | **链接**: [原文](https://blog.codingnow.com/2026/03/star_trek_captains_chair.html)

云风在这篇文章中分享了他对《Star Trek: Captain's Chair》这款2025年桌游的深度体验。这是一款以卡牌构筑为核心玩法的桌游，来自 Nigel Buckle 和 David Turczi 两位设计师，他们之前还创作了《帝国》三部曲。

文章详细分析了游戏的几个核心设计亮点：

**创新的卡组瘦身机制**：
- 可以把卡堆里的牌 LOG 起来：得分卡本身就是 VP，LOG 可以移除不再使用的卡但保留得分
- 卡片可以 deploy 到桌面：提供持久能力的同时让活动卡组瘦身
- 卡片可以 beam 到飞船或星球上：提供更灵活的临时卡组瘦身

**精心设计的卡组升级系统**：
- 每个初始牌组有 5 张固定补充卡和 8 张高级补充卡
- 补充卡通过卡组循环进入抽卡堆，相当于卡组升级
- 调节抽卡堆消耗速度 = 控制玩家卡组升级速度

**控制牌堆轮转的多种方式**：
- 抽牌能力：补充行动选择，加快牌堆轮转
- 弃掉抽牌堆顶：虽然早期被视为负面效果，但配合卡组瘦身可以加快轮转
- 回合结束时保留手牌：让玩家选择确定性或加快轮转
- 获取新牌后自由选择放入弃牌堆或抽牌堆

云风认为这是他这些年玩过的卡牌构筑类桌游中机制、策略和局势变化最丰富的一款。游戏规则复杂（BGG weight 4.06），需要 10 多小时试玩和教学才能掌握，但一旦理解设计逻辑，玩起来颇为流畅。

**推荐理由**：文章对游戏机制的剖析非常深入，展现了资深玩家对桌游设计的独到见解。对于喜欢卡牌构筑和重度策略游戏的玩家来说，这是一篇非常有价值的评测和攻略。

### ⭐⭐⭐⭐⭐ Long-horizon Agent
**作者**: 李乾坤 | **日期**: 2026-02-16 | **链接**: [原文](https://qiankunli.github.io/2026/02/16/long_horizon.html)

李乾坤在这篇文章中探讨了长周期智能体（Long-horizon Agent）的设计理念和实现挑战。长周期任务是指那些需要执行数百甚至数千个步骤才能完成的目标，比如编写复杂软件系统、长期研究项目等。

文章核心观点包括：

**长周期任务的核心挑战**：
- 规划深度问题：随着任务步骤增加，规划树呈指数级增长
- 状态跟踪困难：需要记住大量中间状态和决策点
- 错误传播风险：早期错误会放大到后续所有步骤
- 资源分配优化：如何在多个子任务间合理分配计算资源

**解决方案探索**：
- 分层规划（Hierarchical Planning）：将大任务分解为多层级子任务
- 渐进式验证（Progressive Validation）：每个关键节点都进行中间结果验证
- 快速回滚机制（Fast Rollback）：发现错误时能快速回退到稳定状态
- 知识积累与重用（Knowledge Accumulation）：将成功模式编码为可重用的技能

文章还提出了一个有趣的视角：未来的 AI Agent 可能会从"一次规划多次执行"转向"边规划边学习边执行"的混合模式，更像人类的解决问题方式。

**推荐理由**：这篇文章触及了当前 AI Agent 研究的前沿问题，对于理解和改进长周期任务的执行能力有重要启发意义。作者的观点结合了理论思考和实践经验，值得深入阅读。

### ⭐⭐⭐⭐☆ next prompt工程——skill
**作者**: 李乾坤 | **日期**: 2026-02-07 | **链接**: [原文](https://qiankunli.github.io/2026/02/07/skill.html)

李乾坤在这篇文章中提出了"skill"作为 prompt 工程的一个新抽象概念。他观察到随着模型能力增强，单一 prompt 已经难以覆盖复杂场景，需要更高层次的抽象来管理 prompt 的组合和复用。

**核心观点**：

1. **Skill 的定义**：skill 是可复用、可组合的 prompt 模板，封装了特定任务的最佳实践。类似于编程中的函数或组件。

2. **Skill 的特征**：
   - 可参数化：通过参数调整适应不同场景
   - 可组合：多个 skill 可以组合成更复杂的任务流程
   - 可测试：每个 skill 都有明确的输入输出规范
   - 可版本化：可以追踪和回滚 skill 的历史版本

3. **Skill 的实现方式**：
   ```javascript
   const codeReviewSkill = {
     name: "code_review",
     description: "对代码进行专业审查",
     parameters: {
       language: "编程语言",
       focus: ["security", "performance", "readability"],
       detail_level: "low | medium | high"
     },
     template: (params) => `
       作为一名资深 ${params.language} 工程师，
       请审查以下代码，重点关注：${params.focus.join(', ')}。
       ${params.detail_level === 'high' ? '请提供详细的改进建议和代码示例。' : '请提供简要的评估和关键问题。'}
       \`\`\`${params.language}
       ${params.code}
       \`\`\`
     `
   };
   ```

4. **Skill 生态系统的想象**：
   - Skill 市场：开发者可以分享和发现经过验证的 skill
   - Skill 版本管理：类似 npm 的依赖管理
   - Skill A/B 测试：同一任务的不同 skill 可以对比效果
   - Skill 调优：基于用户反馈自动优化 skill 参数

文章最后指出，skill 抽象不仅提高了 prompt 的可维护性，也为构建更复杂的 AI 应用提供了基础。未来，我们可能会看到类似 "skill-as-code" 的开发范式。

**推荐理由**：这篇文章提出了一个很有前景的抽象概念，对于正在探索 prompt 工程最佳实践的开发者来说，提供了新的思考维度。skill 的概念可能会成为未来 AI 应用开发的重要模式。

### ⭐⭐⭐☆☆ 从chatbot到clawbot
**作者**: 李乾坤 | **日期**: 2026-02-02 | **链接**: [原文](https://qiankunli.github.io/2026/02/02/openclaw.html)

李乾坤在这篇文章中分享了使用 OpenClaw 平台从传统 chatbot 向 clawbot 迁移的经验和思考。OpenClaw 是一个 AI Agent 平台，支持更复杂的任务编排和工具调用能力。

**Chatbot vs Clawbot 的关键区别**：

1. **交互模式**：
   - Chatbot：单轮或简单的多轮对话
   - Clawbot：支持长任务链，可以持续多轮交互

2. **能力边界**：
   - Chatbot：主要依赖模型内置知识
   - Clawbot：可以调用外部工具，实时获取信息

3. **状态管理**：
   - Chatbot：每次请求相对独立
   - Clawbot：维护长期记忆和任务状态

4. **执行方式**：
   - Chatbot：生成文本响应
   - Clawbot：可以执行实际操作（如 API 调用、文件操作）

**迁移经验总结**：

1. **提示词重构**：
   - 从"直接回答问题"转变为"规划执行路径"
   - 增加工具调用的指导
   - 明确错误处理和重试机制

2. **能力定义**：
   - 将自然语言需求拆解为具体工具调用
   - 设计清晰的工具输入输出规范
   - 为工具添加描述性元数据

3. **用户体验优化**：
   - 提供任务进度可视化
   - 支持中途暂停和恢复
   - 添加操作确认机制

文章还分享了一些实用的迁移技巧：
- 从简单任务开始，逐步增加复杂度
- 保持工具集最小化，只保留真正必要的
- 重视错误处理和日志记录

**推荐理由**：这篇文章为想要从传统 chatbot 升级到 AI Agent 的开发者提供了实战经验。作者的观点基于实际项目经验，提出的建议具有可操作性。

### ⭐⭐⭐☆☆ typescript学习
**作者**: 李乾坤 | **日期**: 2026-01-25 | **链接**: [原文](https://qiankunli.github.io/2026/01/25/typescript.html)

李乾坤在这篇学习笔记中记录了他学习 TypeScript 的过程和心得。文章从实用角度出发，重点介绍了 TypeScript 中最常用和最有价值的特性。

**核心学习内容**：

1. **类型系统基础**：
   - 基本类型：string, number, boolean, null, undefined
   - 特殊类型：any, unknown, never, void
   - 类型断言：`as Type` 和 `<Type>value`

2. **函数类型**：
   - 参数类型注解
   - 返回值类型推断
   - 可选参数和默认参数
   - 重载（Overload）的定义和使用

3. **接口与类型别名**：
   - 接口的定义和扩展
   - 类型别名的适用场景
   - 两者何时使用哪个的判断标准

4. **泛型**：
   - 基本泛型语法
   - 泛型约束
   - 泛型工具类型（Partial, Required, Pick, Omit）

**学习心得**：

1. **从 JavaScript 迁移**：
   - 先用 `any` 快速迁移，再逐步精细化类型
   - 利用 IDE 的类型提示逐步完善
   - 不要一次性追求完美类型覆盖

2. **类型设计的哲学**：
   - 类型应该反映业务逻辑，而不仅是数据结构
   - 联合类型（Union Types）比可选类型更精确
   - 使用字面量类型（Literal Types）增强类型安全

3. **实用技巧**：
   - 配置 `strict: true` 获得最佳类型检查
   - 使用 ESLint + TypeScript 双重保护
   - 利用 `d.ts` 文件为第三方库添加类型

**推荐理由**：对于想要从 JavaScript 转向 TypeScript 的开发者，这篇文章提供了一个清晰的学习路径和实用的技巧。作者的学习心得很有参考价值。

### ⭐⭐⭐☆☆ Agent前端
**作者**: 李乾坤 | **日期**: 2026-01-24 | **链接**: [原文](https://qiankunli.github.io/2026/01/24/agent_frontend.html)

李乾坤在这篇文章中探讨了 AI Agent 应用前端设计的挑战和解决方案。与传统应用相比，Agent 应用具有状态不透明、执行时间不确定、结果可能需要多轮迭代等特点，这对前端设计提出了新要求。

**前端设计的关键挑战**：

1. **状态可视化**：
   - Agent 的内部状态（思维链、工具调用、记忆检索）难以直观展示
   - 用户需要理解 Agent 为什么做某个决定

2. **进度反馈**：
   - 任务执行时间不确定，无法准确估计剩余时间
   - 如何展示"正在思考中"的状态

3. **错误处理**：
   - Agent 可能会犯错误，需要让用户感知并允许纠正
   - 错误信息既要准确又不能过于技术化

4. **交互设计**：
   - 如何设计用户与 Agent 的协作界面
   - 什么时候应该询问用户确认，什么时候自主执行

**设计模式和解决方案**：

1. **思维链可视化**：
   - 将 Agent 的推理过程以时间线方式展示
   - 支持折叠/展开，详细/简洁两种视图
   - 用颜色区分不同类型的操作（思考、工具调用、结果获取）

2. **进度指示器设计**：
   - 使用阶段性进度（step x / total steps）而非百分比
   - 提供当前阶段的具体描述
   - 支持用户中途干预和调整

3. **错误恢复机制**：
   - 提供一键重试功能
   - 允许用户手动调整错误的部分
   - 记录错误历史，便于分析模式

4. **协作式交互**：
   - 提供建议供用户确认而非直接执行
   - 支持用户随时插入指导
   - 保留历史对话，方便回溯

**技术实现建议**：

1. **状态管理**：
   - 使用 Redux 或 Zustand 管理复杂的 Agent 状态
   - 实现乐观更新以提升体验
   - 设计清晰的状态机模型

2. **实时通信**：
   - 使用 WebSocket 接收 Agent 的实时更新
   - 实现断线重连和状态同步
   - 考虑使用 SSE（Server-Sent Events）作为简化方案

3. **性能优化**：
   - 使用虚拟滚动处理长对话历史
   - 对思维链使用增量渲染
   - 实现防抖和节流控制更新频率

文章最后强调，Agent 前端不仅是 UI 问题，更是人机交互的重新设计。我们需要让用户既能获得自动化带来的便利，又能保持对过程的感知和控制。

**推荐理由**：这篇文章针对 AI Agent 应用这个新兴领域的前端设计问题提供了系统的思考。对于正在开发 Agent 应用的前端工程师来说，具有很好的参考价值。
